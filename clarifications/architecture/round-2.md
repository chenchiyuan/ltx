# 架构方向 - Round 2

## 大原则确认

### 目标
收敛系统第一阶段的服务边界、任务模式和 GPU 执行模式。

### 范围
- 包含: 对外服务形态、任务提交方式、ComfyUI/LTX Worker 常驻方式、单任务资源占用、ComfyUI 推荐实践约束。
- 不包含: 正式架构文档、代码实现、具体云厂商或 Kubernetes 发行版选型。

---

## 讨论清单

### CLR-ARCH-004: 对外服务形态
- **模糊点**: 第一阶段是内部生产工具还是对外商业化 SaaS。
- **影响范围**: 认证、权限、配额、任务隔离、资产访问和运营后台边界。
- **推荐选项**: 对外 SaaS，但运营后台先做内部可控版。
- **结论**: 第一阶段按对外 SaaS 设计，后台先做内部版。
- **来源分类**: user_confirmed
- **状态**: 生效
- **确认时间**: 2026-07-13

### CLR-ARCH-005: 任务交互方式
- **模糊点**: 用户提交任务后是同步等待还是异步排队。
- **影响范围**: API 设计、前端进度展示、队列、重试、取消和结果通知。
- **推荐选项**: 异步排队。
- **结论**: 第一阶段采用异步排队任务系统。
- **来源分类**: user_confirmed
- **状态**: 生效
- **确认时间**: 2026-07-13

### CLR-ARCH-006: GPU Worker 生命周期
- **模糊点**: ComfyUI/LTX 2.3 是长期常驻还是按任务动态启动。
- **影响范围**: 模型加载成本、失败率、GPU 利用率、服务发现。
- **推荐选项**: 常驻 Worker 模式。
- **结论**: 第一阶段采用常驻 Worker 模式。
- **来源分类**: user_confirmed
- **状态**: 生效
- **确认时间**: 2026-07-13

### CLR-ARCH-007: 单任务资源粒度
- **模糊点**: 第一阶段是否做多卡切分。
- **影响范围**: 调度复杂度、任务隔离、计数和失败恢复。
- **推荐选项**: 单任务单卡。
- **结论**: 第一阶段按单任务单卡设计。
- **来源分类**: user_confirmed
- **状态**: 生效
- **确认时间**: 2026-07-13

### CLR-ARCH-008: ComfyUI 实践约束
- **模糊点**: GPU 服务发现和执行架构是否直接按 Kubernetes 假设推进。
- **影响范围**: 执行平面设计、部署方式、ComfyUI API 集成方式。
- **推荐选项**: 先研究 ComfyUI 推荐实践，再据此确定架构。
- **结论**: GPU 执行架构需要以 ComfyUI 官方推荐实践和 LTX 官方 ComfyUI 工作流实践为重要约束。
- **来源分类**: user_confirmed
- **状态**: 生效
- **确认时间**: 2026-07-13
