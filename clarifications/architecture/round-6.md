# 架构方向 - Round 6

## 大原则确认

### 目标
收敛第一阶段对外 API 隔离、额度/价格记录、上线观测指标、容量策略和工作流发布流程。

### 范围
- 包含: API Key 隔离、账号与 Key 的后续关系、额度/计费记录口径、Worker 指标、静态容量池、工作流发布状态。
- 不包含: 组织/团队权限实现、正式价格表、自动扩容实现、代码实现。

---

## 讨论清单

### CLR-ARCH-024: API 资源隔离方式
- **模糊点**: 第一阶段多租户隔离基于账号、组织还是 API Key。
- **影响范围**: 认证、授权、资源隔离、任务归属、后续组织权限演进。
- **推荐选项**: 对外只提供 API，通过 API Key 做资源隔离；账号与 Key、组织/团队关系后续再做。
- **结论**: 第一阶段对外只提供 API，通过 API Key 做资源隔离。后续基于账号和 Key 的关系做组织/团队，但当期不做。
- **来源分类**: user_confirmed
- **状态**: 生效
- **确认时间**: 2026-07-13

### CLR-ARCH-025: 额度与价格记录口径
- **模糊点**: 第一阶段是否直接做复杂价格系统。
- **影响范围**: 计费准确性、成本分析、后续价格表设计、任务审计。
- **推荐选项**: 先按任务档位 + GPU 消耗量组合记录，价格系统后置。
- **结论**: 计费或额度先按任务档位 + GPU 消耗量组合记录，不在第一版做复杂价格系统。记录 profile、estimated_gpu_seconds、actual_gpu_seconds、attempt_count、success/failure。
- **来源分类**: user_confirmed
- **状态**: 生效
- **确认时间**: 2026-07-13

### CLR-ARCH-026: Worker 级可观测性
- **模糊点**: Worker 级指标是否作为上线硬要求。
- **影响范围**: 故障诊断、容量规划、生成性能分析、异常 Worker 发现。
- **推荐选项**: 把 GPU 利用率、显存、队列长度、任务成功率、失败原因、平均生成耗时作为上线硬要求。
- **结论**: Worker 级指标作为上线硬要求，至少包含 GPU 利用率、显存、队列长度、任务成功率、失败原因、平均生成耗时。
- **来源分类**: user_confirmed
- **状态**: 生效
- **确认时间**: 2026-07-13

### CLR-ARCH-027: 第一阶段容量策略
- **模糊点**: 第一阶段是否做按需自动扩容 GPU 服务器。
- **影响范围**: 部署复杂度、模型预热、成本控制、节点注册、容量稳定性。
- **推荐选项**: 第一阶段采用静态容量池，不做按需自动扩容 GPU 服务器。
- **结论**: 第一阶段采用静态容量池，不做按需自动扩容 GPU 服务器。
- **来源分类**: user_confirmed
- **状态**: 生效
- **确认时间**: 2026-07-13

### CLR-ARCH-028: 工作流发布流程
- **模糊点**: 工作流修改是否直接影响线上任务。
- **影响范围**: 产出稳定性、失败率控制、回滚、灰度和后台操作安全。
- **推荐选项**: 采用草稿 -> 测试 -> 发布 -> 回滚的版本流程。
- **结论**: 工作流发布采用草稿 -> 测试 -> 发布 -> 回滚的版本流程。
- **来源分类**: user_confirmed
- **状态**: 生效
- **确认时间**: 2026-07-13
